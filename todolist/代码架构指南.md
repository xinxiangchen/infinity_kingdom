# 代码架构指南

更新日期：2026-06-09  
当前依据：本地 `master` 已从 `origin/master` 更新到 `d6b7b2f Fix startup errors and polish combat UI`。

这份文档给整合组、角色组、地图组、UI 组和后续 C++/GDExtension 组共用。核心目标是：各模块封装自己的状态和实现细节，只暴露清晰 API、信号和数据结构，主流程只负责调度，不深入改每个模块内部。

## 1. 当前总体结构

当前项目是一个 Godot 4 的 2D 动作 Roguelite 原型。运行入口在 `app_entry.tscn` / `app_entry.gd`，主流程在 `world.tscn` / `world.gd`。`world.gd` 目前承担了较多职责：选人、开局、地图房间切换、战斗生成、奖励、事件、HUD、音频、结果界面都由它串起来。

现阶段能跑通的主循环是：

1. 进入标题/选人界面。
2. 选择正常模式或角色调试模式。
3. 正常模式中选择骑士、游侠或法师。
4. 进入饰品选择或开局准备。
5. `MapRuntime` 根据地图素材生成横向房间、碰撞、入口出口、随机物件和房间锁定摄像机。
6. `world.gd` 按遭遇顺序生成普通小怪、守关 Boss、空房间、双子 Boss 和最终 Boss。
7. 每场战斗结束后进入金币、经验、饰品、事件或服务节点。
8. 玩家死亡或胜利后进入结果界面，并可重开。

重要判断：项目已经有“可试玩整合形态”，但架构仍处在“主流程集中调度”的阶段。下一步应把 `world.gd` 中的长期规则拆到独立系统里，让主流程只连接系统之间的信号。

## 2. 目录职责

| 目录/文件 | 当前职责 | 后续边界建议 |
|---|---|---|
| `app_entry.gd` / `app_entry.tscn` | Godot 启动入口，进入主程序 | 只负责启动场景和启动参数，不承载玩法 |
| `world.gd` / `world.tscn` | 主流程整合、选人、地图、战斗、UI、奖励、音频 | 保留为“流程编排器”，逐步把具体规则下沉到系统模块 |
| `characters/` | 三个玩家角色及状态机 | 每个角色只暴露统一角色 API，不直接依赖地图、UI、奖励系统 |
| `actors/enemy/` | 普通敌人与六类小兵逻辑 | 小兵数据表化，敌人只管行为与战斗，不管刷怪流程 |
| `actors/bosses/` | Boss 场景和脚本 | Boss 暴露统一 Boss API，阶段和特殊机制内部封装 |
| `actors/encounters/` | 遭遇场景，负责生成本场敌人并发出清场信号 | 遭遇只管本房间敌人波次，不管下一张图和奖励 |
| `combat/` | 伤害、生命、运行期贴图、通用战斗工具 | 后续优先迁移到 C++ 的候选层 |
| `systems/map/` | 地图运行期拼接、房间碰撞、摄像机锁定、随机物件 | 地图组主要负责这里和地图工具 |
| `systems/run/` | 周目状态、金币、事件、奖励修正、音频路线 | 后续拆出血脉/代际/章节进度系统 |
| `systems/accessories/` | 饰品数据、选择、装备效果 | 保持数据驱动，不让角色直接写饰品规则 |
| `systems/pickups/` | 拾取物、掉落、运行期拾取逻辑 | 后续可扩展消耗品槽 |
| `ui/` | 选人、模式选择、战斗状态、事件面板、设置、调试 UI | UI 只读状态、发请求，不直接改玩法数据 |
| `audio/` | 音乐管理和音量设置 | 通过 profile/route 播放，不让 Boss 或地图直接切 BGM |
| `tools/` | 地图预览、角色调试、Boss 预览、导出脚本 | 只作为开发工具，不进入正式主流程规则 |
| `src/`、`thirdparty/godot-cpp/` | C++/GDExtension 预留和依赖 | 后续放高频逻辑，GDScript 保留编排 |
| `docs/` | 已有协作文档和代码规范 | 长期文档保留这里；临时待办和差距分析放 `todolist/` |

## 3. 主流程模块

### 3.1 `world.gd` 的当前职责

`world.gd` 是当前整合中心，主要做这些事：

- 预加载玩家角色、遭遇、Boss、地图运行期、拾取物和 UI 脚本。
- 连接选人、饰品选择、事件面板、暂停、结果、设置等 UI 信号。
- 创建并激活 `MapRuntime`。
- 维护当前 `encounter_index` 和 `current_encounter`。
- 根据 `ENCOUNTER_SCENES` 与 `FINAL_BOSS_SCENES` 生成敌人。
- 控制房间入口出口、自动走门、黑幕转场和摄像机更新。
- 调用 `RunDirector` 奖励金币、经验、事件路线。
- 监听角色受击、敌人死亡、战斗结束，刷新 HUD 和世界血条。
- 调用 `Music` / `AudioRoute` 播放对应阶段 BGM。

这部分当前可用，但耦合偏重。后续建议拆成：

| 目标系统 | 从 `world.gd` 拆走的内容 |
|---|---|
| `EncounterFlow` | 遭遇索引、生成遭遇、清场判断、进入下一房间 |
| `RoomTransitionService` | 自动走门、黑幕、入口出口、转场锁输入 |
| `RunProgression` | 胜利/失败、章节推进、代际重开、皇帝血量持久化 |
| `RewardFlow` | 金币、经验、饰品、事件、服务节点触发顺序 |
| `WorldHudBinder` | HUD 绑定、血条创建、UI 状态刷新 |
| `AudioRouteBinder` | 根据房间、Boss、结果切 BGM |

### 3.2 主流程对外 API 建议

主流程只应该提供少量高层入口：

```gdscript
func start_normal_run(hero_id: String) -> void
func start_debug_run(hero_id: String, enemy_id: String) -> void
func request_next_room() -> void
func report_encounter_cleared(summary: Dictionary) -> void
func report_player_dead(summary: Dictionary) -> void
func restart_to_title() -> void
```

其他模块不要直接改 `encounter_index`、不要直接 `queue_free()` 主流程节点、不要直接切 `Music`。需要影响主流程时，用信号或上述 API。

## 4. 角色模块

### 4.1 当前角色结构

`characters/knight/`、`characters/ranger/`、`characters/mage/` 各自有场景、主脚本和状态机。它们已经比早期“贴图直接移动”更接近模块化：角色内部处理移动、攻击、技能、资源、受击、动画/贴图姿态。

角色调试入口在 `tools/character_debug_world.tscn`，UI 在 `ui/character_debug_status.tscn` 与 `ui/debug_enemy_select.tscn`，可以脱离地图直接测试角色和敌人。

### 4.2 角色组必须统一暴露的 API

每个玩家角色建议实现同一组方法。主程序只认这些方法，不关心角色内部是状态机、动画树还是未来 C++ 组件。

```gdscript
func setup_run_context(context: Dictionary) -> void
func apply_run_modifiers(modifiers: Dictionary) -> void
func apply_accessory_effect(effect: Dictionary) -> void
func clear_run_effects() -> void

func get_display_name() -> String
func get_hud_state() -> Dictionary
func get_combat_state() -> Dictionary
func get_facing_direction() -> int

func set_control_enabled(enabled: bool) -> void
func set_auto_walk_direction(direction: Vector2) -> void
func clear_control_effects(clear_movement: bool, clear_attack: bool, clear_skill: bool) -> void

func receive_damage(payload: Dictionary) -> Dictionary
func heal(amount: float) -> void
func restore_resource(kind: String, amount: float) -> void
func is_alive() -> bool
```

建议统一信号：

```gdscript
signal died(summary: Dictionary)
signal health_changed(current: float, maximum: float)
signal resource_changed(kind: String, current: float, maximum: float)
signal attack_started(payload: Dictionary)
signal attack_resolved(payload: Dictionary)
signal skill_used(payload: Dictionary)
signal state_changed(state: Dictionary)
```

### 4.3 角色组不要做的事

- 不要在角色脚本里直接引用 `world.gd`。
- 不要直接打开 UI 面板。
- 不要自己决定地图进度、掉落、金币、BGM。
- 不要把敌人或 Boss 的特殊逻辑写进玩家角色。
- 不要为了某个角色单独新增主流程分支，除非先抽象成通用能力。

角色可以读取 `context`，例如当前房间、构筑效果、调试模式，但只能把它当配置，不应把整合流程写进自己内部。

## 5. 敌人与 Boss 模块

### 5.1 普通敌人

当前普通小怪集中在 `actors/enemy/town_enemy.gd`，可表现六类基础兵种。后续建议把“兵种参数”数据表化：

```gdscript
{
	"id": "knight_swordsman",
	"display_name": "持剑卫兵",
	"family": "knight",
	"scene": "res://actors/enemy/town_enemy.tscn",
	"sprite": "...",
	"weapon": "...",
	"hp": 90,
	"move_speed": 120,
	"attack_profile": "melee_slash"
}
```

遭遇场景只按 `enemy_id` 生成，不直接关心贴图和行为参数。

### 5.2 Boss

Boss 当前分布在 `actors/bosses/town/`。有守关 Boss、双子 Boss、多个最终 Boss。第七章废案列阵机制已从流程中移除，代码保留在仓库中，主流程使用 `empty_encounter.tscn`。

Boss 建议统一暴露：

```gdscript
func setup_boss(context: Dictionary) -> void
func start_battle(player: Node) -> void
func set_battle_enabled(enabled: bool) -> void
func get_boss_state() -> Dictionary
func get_music_profile() -> StringName
```

统一信号：

```gdscript
signal boss_defeated(summary: Dictionary)
signal phase_changed(phase_id: String, payload: Dictionary)
signal boss_health_changed(current: float, maximum: float)
```

Boss 自己可以处理阶段、武器、朝向、动画节奏和弹幕，但不要自己进入下一关、发奖励或切场景。

## 6. 地图模块

### 6.1 当前地图运行期

`systems/map/map_runtime.gd` 会从 `tools/map_browser_demo.gd` 的地图配置读取房间素材，横向拼成一串房间。它当前负责：

- 加载地图大图。
- 计算房间矩形和可行走矩形。
- 给每张图添加围墙碰撞。
- 在左右边缘添加入口/出口视觉标记。
- 随机放置每张图对应的小物件。
- 给物件生成基于透明像素外框/多边形的碰撞。
- 添加房间锁定摄像机，让镜头只在当前房间范围内移动。

### 6.2 地图组 API 建议

地图模块对主流程暴露：

```gdscript
func setup(world_root: Node2D, spawn_marker: Marker2D, encounter_marker: Marker2D, rng: RandomNumberGenerator) -> void
func build() -> void
func activate_room(room_index: int, player_character: Node) -> void
func update_camera(room_index: int, player_character: Node, force: bool = false) -> void
func player_spawn_for_room(room_index: int) -> Vector2
func encounter_spawn_for_room(room_index: int) -> Vector2
func room_entrance_target(room_index: int) -> Vector2
func room_exit_target(room_index: int) -> Vector2
```

地图组不要直接生成敌人，也不要决定这一房间打什么怪。地图只告诉主流程：玩家站哪、敌人生成点在哪、镜头如何锁定、碰撞在哪里。

### 6.3 地图碰撞规范

- 地图本体碰撞只围住不可通过的墙、边界、建筑边缘，不要围住草地、道路或主可行走区域。
- 每张图的碰撞应按视觉检查逐图调整，不能只用一个固定比例套所有图。
- 小物件碰撞用素材非透明像素的最高、最低、最左、最右外框作为基础框，再按体验缩小 5%-15%，避免玩家被视觉外透明边卡住。
- 可挡子弹/攻击的小物件需要加入 `projectile_blocker` 组，并设置合理 `collision_layer` / `collision_mask`。
- 随机物件必须避开入口、出口、玩家出生点、敌人出生点和已有物件。

## 7. UI 模块

UI 当前包括：

- `ui/character_select.*`：选人。
- `ui/play_mode_select.*`：正常/调试模式选择。
- `ui/battle_status.*`：战斗状态和流程提示。
- `ui/knight_hud.*`、`ui/hud_meter_bar.gd`、`ui/cooldown_skill_icon.gd`：战斗 HUD。
- `ui/accessory_choice.*`：饰品选择。
- `ui/run_event_panel.*`：事件/商店/服务选择。
- `ui/debug_enemy_select.*`、`ui/character_debug_status.*`：角色调试。
- `ui/pause_menu.*`、`ui/result_screen.*`、`ui/settings_panel.*`、`ui/audio_settings_panel.*`：系统 UI。

UI 的边界：

- UI 发出 `xxx_requested` 或 `xxx_selected` 信号。
- UI 可以展示 `Dictionary` 状态。
- UI 不直接改角色数值、金币、房间、BGM。
- UI 文案应支持简体中文、繁体中文、英文三套，至少新增中文时不要破坏已有英文键。

## 8. 运行进度、奖励和饰品

### 8.1 `RunDirector`

`systems/run/run_director.gd` 当前维护：

- 金币。
- 已清遭遇数。
- 事件路线。
- 本局修正。
- 下一场临时准备效果。
- 事件历史、奖励历史。
- 英雄等级、经验、击杀数。
- 城镇服务节点是否已使用。

这是后续“血脉火种、代际继承、皇帝持久血量”的自然扩展点，但不建议继续把所有规则塞进一个单例。建议拆成：

| 系统 | 职责 |
|---|---|
| `RunDirector` | 单局进度、金币、事件路线、房间奖励 |
| `LineageDirector` | 血脉火种、代际编号、资质继承、死亡重生 |
| `EmperorProgress` | 皇帝跨代剩余血量、本大周目胜负 |
| `BuildDirector` | 技能分支、武器强化、饰品、消耗品槽 |

### 8.2 饰品与构筑

`systems/accessories/accessory_manager.gd` 当前适合作为饰品数据和效果入口。建议继续保持：

- 饰品数据集中注册。
- 选择逻辑由管理器生成候选。
- 角色只接收最终效果，不关心候选来源。
- UI 只显示候选，不自己计算效果。

## 9. 音频模块

音频应统一通过 `audio/music_manager.gd` 和 `systems/run/audio_route.gd` 管理。推荐规则：

- 地图、Boss、事件只返回 `music_profile` 或阶段标签。
- `world.gd` 或后续 `AudioRouteBinder` 决定何时切换。
- 不要在 Boss 脚本里直接 `Music.play_profile()`，否则转场、暂停和结果界面会打架。
- 守关 Boss、王子/双子 Boss、最终 Boss 应使用不同 profile，避免同一首音乐误用。

## 10. C++/GDExtension 迁移边界

当前仓库有 `thirdparty/godot-cpp/` 和 `SConstruct`，说明已经具备后续 GDExtension 的基础依赖。但当前主流程仍以 GDScript 为主，且 C++ 扩展不应一次性吞掉整套玩法。

建议迁移顺序：

1. 先迁移纯计算、无场景依赖、可单元测试的逻辑。
2. 保持 GDScript API 不变，让主流程调用同名包装方法。
3. 每迁移一个模块都保留 GDScript fallback 或测试场景。

优先候选：

| 候选 | 原因 |
|---|---|
| 伤害结算 payload 解析 | 高频、纯计算、容易测试 |
| 碰撞/像素外框生成 | 地图物件多时可能偏重 |
| 敌人目标选择与转向 | 高频、规则固定 |
| 弹幕/投射物批量更新 | 数量多时收益明显 |
| 构筑修正合并 | 纯数据合并，便于保证一致性 |

不建议首批迁移：

- UI。
- 选人和菜单流程。
- 地图场景拼装。
- 仍在频繁改规则的 Boss 技能。

## 11. 模块协作规则

### 11.1 新功能提交前要回答的问题

1. 这个功能属于哪个模块？
2. 它需要暴露给谁调用？
3. 暴露的是方法、信号还是数据表？
4. 有没有直接引用不该引用的模块？
5. 调试入口能不能单独测？
6. 改动后正常模式和调试模式是否都还能进？

### 11.2 推荐依赖方向

推荐：

```text
world/flow -> systems -> actors/characters -> combat
world/flow -> ui
world/flow -> audio
world/flow -> map
ui -> signals -> world/flow
```

避免：

```text
characters -> world
boss -> ui
map -> encounter
ui -> RunDirector direct mutation
enemy -> Music
```

### 11.3 数据结构约定

跨模块传值尽量用 `Dictionary`，但必须固定键名。示例：

```gdscript
var damage_payload := {
	"amount": 24.0,
	"source": self,
	"source_id": "knight_slash",
	"damage_type": "physical",
	"can_crit": true,
	"knockback": Vector2.ZERO,
	"tags": ["melee", "weapon"]
}
```

不要让每个角色、敌人、Boss 都发不同格式的 payload。字段不确定时，先在文档里补，再写代码。

## 12. 测试与验收

推荐每次合入前至少做：

1. Godot 打开项目无红色解析错误。
2. 正常模式能从选人进入第一张图。
3. 第一张图清完一轮怪后能进入下一张图。
4. 守关 Boss 有形象、血条、武器、BGM。
5. 双子 Boss 有形象、血条、武器、BGM。
6. 角色调试模式能选择角色和测试敌人。
7. 暂停、结果、音量设置能打开关闭。
8. 地图碰撞不会挡住主路，墙体能挡住玩家。

若改到地图或 Boss 动画，必须实际打开试玩，不能只看代码。

## 13. 下一步解耦建议

优先级从高到低：

1. 把 `world.gd` 的遭遇生成、清场和下一房间逻辑拆到 `systems/run/encounter_flow.gd`。
2. 新建 `systems/run/lineage_director.gd`，承接血脉火种和代际重生。
3. 新建 `systems/run/emperor_progress.gd`，承接皇帝跨代血量。
4. 新建 `systems/interaction/interaction_service.gd`，统一拾取、商店、教堂、军需库、准备点优先级。
5. 把角色统一 API 写成文档和测试脚本，角色组后续按 API 改。
6. 把敌人/Boss 参数逐步表格化，减少主流程硬编码。
7. C++ 先做伤害结算或地图物件碰撞生成，不要先碰 UI 和流程。
