更新日期：2026-06-09

这份文档用于说明：当前代码已经完成了哪些可试玩内容，相比策划案还缺哪些系统，以及后续各模块应该按什么顺序补。

## 1. 总体结论

当前项目已经不是纯 Demo，而是一个能跑主流程的可试玩原型：有三职业、地图房间、普通小怪、多个 Boss、房间转场、随机地图物件、HUD、事件/饰品/金币/经验、音频和角色调试入口。

但它还没有完全达到策划案的核心卖点。策划案最核心的“父死子继”“血脉火种”“皇帝血量跨代保留”“资质继承”还没有形成完整系统。目前更像“一次性通关的房间制动作 Roguelite 原型”，而不是完整的“轮回王国”。

后续要优先补的是大周目/小轮回结构，而不是继续堆更多 Boss 或更多饰品。

## 4. P0 待办：先补核心卖点

P0 是决定项目是不是“轮回王国”的系统，应先做。

### 4.0 王冠循环与多结局

目标：

- 打败王子/双子后不直接结束游戏，而是进入下一轮皇帝循环。
- 下一轮只能从剩余两个家族中选择角色。
- 再击败一任皇帝后，只能从最后一个未登基家族中选择角色。
- 三个家族都曾当过皇帝后，进入最终结局分歧。
- 当前皇帝家族决定普通小兵刷新权重：皇帝家族小兵 50%，另外两家各 25%。
- 结局至少包括：打碎王冠、逃向远方、火种熄灭、戴上王冠 bad ending。

建议新增：

- `systems/run/throne_cycle_director.gd`
- `ui/ending_choice_panel.gd`

验收标准：

- 王子战胜利后不会直接显示最终结局。
- 可选角色会随着已登基家族减少。
- 小兵刷新会受当前皇帝家族影响。
- 四类结局都能被明确触发。

### 4.1 血脉火种系统

目标：

- 大周目开始时火种为 5。
- 玩家死亡消耗 1 枚火种。
- 火种还剩时进入下一代生成流程。
- 火种耗尽且皇帝未死则大周目失败。
- UI 显示当前火种、当前代数、死亡后的扣除结果。

建议新增：

- `systems/run/lineage_director.gd`
- `ui/lineage_panel.gd`
- `ui/lineage_panel.tscn`

核心 API：

```gdscript
func start_lineage(family_id: String) -> void
func can_respawn() -> bool
func consume_seed(death_summary: Dictionary) -> Dictionary
func generate_next_heir(parent_summary: Dictionary) -> Dictionary
func get_lineage_state() -> Dictionary
```

验收标准：

- 进入新大周目显示 5 枚火种。
- 死亡后显示剩余 4 枚，并重新进入当前章节入口。
- 火种归零后进入失败结果。
- 重开大周目后火种恢复 5。

### 4.2 资质与继承

目标：

- 每一代有强壮、迅捷、专注三维资质。
- 总点数 7-15。
- 第一代开局应有资质分配或微调入口，不应固定 3/3/3 后直接进游戏。
- 下一代总点数由上一代评分修正：S +3，A +2，B +1，C +0，D -1。
- 玩家可微调 1 点：从一个属性挪到另一个属性，不额外增加总点数。
- 资质转换为生命、防御、移速、攻速、冷却、攻击/灵感、暴击等属性。

建议新增：

- `systems/run/aptitude_generator.gd`
- `ui/heir_select_panel.gd`

验收标准：

- 开局出现第一代资质分配或微调面板。
- 死亡后出现下一代资质面板。
- 玩家能看到父代表现如何影响下一代。
- 微调 1 点后进入当前章节入口，且总点数仍保持 7-15。
- 三职业应用同一资质规则，但映射到职业核心属性不同。

### 4.3 皇帝血量跨代保留

目标：

- 皇帝有大周目级最大血量和剩余血量。
- 同一大周目内，玩家对皇帝造成的伤害会保存。
- 玩家死亡后，下一代再次进入皇帝战时从剩余血量开始。
- 首版只保存血量，不保存阶段、仇恨、临时状态。

建议新增：

- `systems/run/emperor_progress.gd`

核心 API：

```gdscript
func start_emperor(max_hp: float) -> void
func apply_emperor_damage(amount: float) -> void
func get_remaining_hp() -> float
func get_progress_state() -> Dictionary
func is_emperor_defeated() -> bool
```

验收标准：

- 第一代把皇帝打到 70% 后死亡。
- 第二代进入皇帝战，皇帝从 70% 开始。
- 皇帝死亡后本大周目胜利。
- 重开大周目后皇帝血量重置。

### 8.6 数值与策划组

- 定义三维资质到属性的映射。
- 定义武器词条和消耗品池；技能分支第一版暂不进入。
- 定义消耗品池和价格。
- 定义普通兵权重和皇宫精英参数。

### 8.7 C++/GDExtension 组

- 先保证 Godot 能加载最小扩展。
- 从纯计算模块开始迁移。
- 每个 C++ 类提供 GDScript 可调用包装。
- 写清楚构建命令、输出路径和不要提交的中间文件。

## 9. 推荐开发顺序

建议后续两轮先按这个顺序做：

1. 整合组新增 `LineageDirector` 和 `EmperorProgress` 的 GDScript 版本。
2. UI 组补火种、代际、皇帝剩余血量显示。
3. 角色组统一三职业 API，保证死亡 summary 和战斗 summary 能返回。
4. 整合组把死亡后重生接到当前章节入口。
5. 数值组补资质生成和属性映射。
6. UI 组补资质微调界面。
7. 地图组完成逐图碰撞复核。
8. 敌人组完成普通兵数据表和权重。
9. 构筑组补武器强化、消耗品槽；技能分支第一版暂不做。
10. C++ 组开始迁移伤害结算或碰撞外框检测。

## 10. 最小验收版本定义

达到下面条件，才算更接近策划案里的 MVP：

- 选择家族后，整个大周目固定该职业。
- 玩家有 5 枚血脉火种。
- 死亡扣 1 枚火种并生成下一代。
- 下一代有继承资质，能微调 1 点。
- 皇帝血量在同一大周目内不会回满。
- 城镇和皇宫至少各有一组明确房间流程。
- 第一版不要求技能分支成长，但现有技能必须稳定可用。
- 每代有固定武器强化、一个首饰槽、一个消耗品槽。
- 存档能保存王冠循环、皇帝血量、已登基家族、当前代构筑和服务节点状态。
- 教堂、军需库、商人、皇帝门前准备点至少以可交互节点形式存在。
- 角色调试入口和正常模式都能运行。

当前项目已经有骨架和不少可玩内容，下一步要把“轮回”从文档里的设定变成代码里的主循环。
