# 教堂、商店、军需库交互节点待办

更新日期：2026-06-11

本文档用于固定后续场景交互节点设计。当前项目里很多服务被塞进了关间三选一卡牌系统，例如 `services`、`shop`、`rest`、`forge`。后续要把这些内容从“每关后随机/半随机事件”迁移成地图上的固定交互节点。

## 1. 总体目标

后续地图中应接入三个明确的服务节点：

```text
教堂
商店
军需库
```

它们应作为地图上的可交互对象出现，而不是每过一关都自动弹三选一奖励。

玩家靠近节点后按交互键打开对应 UI。UI 可以复用现在的卡牌样式，但系统逻辑要拆清楚。

## 2. 三个节点的定位

| 节点 | 核心定位 | 资源消耗 | 是否应随机 |
|---|---|---|---|
| 教堂 | 恢复、净化、稳定状态 | 免费或少量金币 | 不随机，固定交互 |
| 商店 | 花钱购买饰品、武器升级、状态 | 消耗金币 | 商品可随机，节点固定 |
| 军需库 | 武器升级或免费装备补给 | 免费或有限次数 | 选项可随机，节点固定 |

## 3. 教堂

### 3.1 功能定位

教堂主要负责恢复和保命，不负责武器成长。

可提供：

```text
恢复生命
恢复护甲
恢复灵感
清除负面状态
少量临时防护
```

### 3.2 第一版建议选项

教堂打开后可以三选一：

```text
医疗包：恢复 45% 生命
守护蜡烛：恢复护甲与灵感
野战修整：恢复护甲，并提升少量最大生命
```

这些可以沿用当前 `RunEffects` 里的：

```text
rest_heal
rest_focus
rest_repair
```

但后续触发来源要从关间事件改成地图交互节点。

### 3.3 使用次数

建议每个教堂节点每代只能使用一次。

```text
同一代用过后，该节点变成已使用状态。
死亡轮回后，节点使用状态重置。
```

## 4. 商店

### 4.1 功能定位

商店负责金币消费。后续商店应提供：

```text
饰品
花钱升级武器
购买状态
```

### 4.2 第一版商品类型

建议商品分三类：

```text
饰品商品：购买后打开饰品选择或直接获得一件饰品
武器升级商品：花金币获得一次武器升级选择
状态商品：购买一项本代/本局状态强化
```

### 4.3 当前已有可迁移内容

当前代码里已有这些商店选项：

```text
shop_attack：花 28 金币，本局攻击伤害 +10%
shop_defense：花 26 金币，恢复护甲并提升最大护甲
shop_relic：花 36 金币，获得一次额外饰品选择
```

后续应调整成：

```text
shop_relic -> 商店饰品
shop_weapon_upgrade -> 花钱升级武器
shop_status_attack / shop_status_defense -> 买状态
```

### 4.4 商店商品数量

建议每次商店展示 3-4 个商品：

```text
至少 1 个饰品相关
至少 1 个武器升级相关
至少 1 个状态相关
第 4 格可随机补给/折扣/高价值商品
```

玩家可以买多个，还是只能买一个，需要后续决定。第一版建议：

```text
每个商店节点每代最多购买 1 次
```

这样方便控制强度，也减少 UI 和经济复杂度。

## 5. 军需库

### 5.1 功能定位

军需库负责免费或低成本军备补给。它和商店区别是：

```text
商店 = 花钱买
军需库 = 免费拿一次，但选择更受限制
```

军需库可以提供：

```text
武器升级
免费装备
护甲/补给
短期战斗准备
```

### 5.2 第一版建议选项

军需库打开后三选一：

```text
武器维护：获得一次武器升级
轻甲补给：恢复护甲并提升少量最大护甲
战备装备：获得一件免费装备或临时状态
```

其中“武器维护”应接入独立武器升级系统，不再使用当前 `forge_*` 状态奖励冒充武器升级。

### 5.3 免费装备的定义

“免费装备”第一版可以先做成轻量规则，不做完整背包：

```text
临时护甲
一次性护盾
短期攻击/移速状态
免费消耗品
```

如果后续装备系统来不及做完整，可以把免费装备先做成“装备卡/状态卡”，但 UI 文案要明确它不是饰品，也不是武器升级。

## 6. 和现有三选一系统的迁移关系

当前已有：

```text
services：一次性三选一，里面包含教堂、军需库、商店推荐卡
shop：完整商店事件，但不在普通事件池中
rest：教堂恢复事件
forge：锻造状态奖励
```

后续迁移目标：

```text
services -> 删除或只作为旧流程兼容
rest_* -> 迁移到教堂节点
shop_* -> 迁移到商店节点
forge_* -> 部分废弃，部分迁移为军需库状态补给
武器升级 -> 新建独立 WeaponUpgradeManager
```

不要继续把所有服务塞进过关后的随机事件池。

## 7. 推荐交互流程

### 7.1 地图上

```text
玩家进入服务房间
看到教堂 / 商店 / 军需库节点
靠近节点
出现交互提示
按交互键打开对应 UI
选择后节点标记为已使用
继续前进
```

### 7.2 UI 层

可以复用现有卡牌 UI 风格，但建议拆成独立入口：

```text
ui/church_panel.gd
ui/shop_panel.gd
ui/armory_panel.gd
ui/weapon_upgrade_panel.gd
```

如果时间不够，也可以先复用 `run_event_panel.gd`，但调用时要传入明确的节点类型，不要继续走随机事件池。

## 8. 数据结构建议

服务节点统一数据：

```gdscript
{
	"id": "shop_01",
	"type": "shop",
	"used": false,
	"room_index": 5,
	"choices": []
}
```

商品数据：

```gdscript
{
	"id": "shop_weapon_upgrade",
	"type": "weapon_upgrade",
	"name": "武器升级",
	"cost": 45,
	"summary": "花费金币，选择一项武器强化。",
	"payload": {
		"open_panel": "weapon_upgrade"
	}
}
```

军需库选项：

```gdscript
{
	"id": "armory_weapon_maintenance",
	"type": "weapon_upgrade",
	"name": "武器维护",
	"cost": 0,
	"summary": "免费获得一次武器升级。",
	"payload": {
		"open_panel": "weapon_upgrade"
	}
}
```

## 9. 验收标准

- 地图上能看到教堂、商店、军需库节点。
- 玩家靠近后能交互打开对应 UI。
- 教堂能恢复状态。
- 商店能购买饰品、花钱升级武器、购买状态。
- 军需库能选择武器升级或免费装备。
- 节点使用后不会反复无限领取。
- 轮回后节点状态按设计重置。
- 普通过关不再强制弹出大量状态/饰品奖励。
